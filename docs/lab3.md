# Lab3: CQS (Command-Query Separation). Analysis and Comparation

---

### 1. Що змінилося в структурі проєкту порівняно з Лабораторною №2?

Основна зміна полягає в **декомпозиції Application Layer**. Якщо раніше сервіси являли собою перевантажені god-like класи, які містили у собі усю логіку оркерстрації, то тепер:

* **Commands:** Кожна дія, що змінює стан системи (Create, Update, Delete), винесена в окремий клас-команду.
* **Queries:** Кожна операція читання винесена в окремий запит.
* **Handlers:** З'явилися специфічні класи-обробники (handlers), кожен з яких відповідає лише за одну операцію та один UseCase.
* **Bus:** Впроваджено `CommandBus` та `QueryBus` (через `@nestjs/cqrs`), які виступають посередниками між `Presentation` та `Application`.


---

### 2. Переваги CQS: чим допомогло розділення?

1.  **Дотримання SRP (Single Responsibility Principle):** Кожен хендлер відповідає за один конкретний Use Case. Це виключає появу так званих "god services", де в кучу звалено безліч операцій
2. **Масштабованість:** Можливість незалежно масштабувати читання та запис. Наприклад, спрямувати всі Query Handlers на репліку БД для читання, що критично для високонавантажених систем. Це стосується більше CQRS, але таке розділення на команди и записи - перший крок до цієї архітектури
3.  **Покращена тестованість:** Тестувати один метод `execute()` у маленькому хендлері набагато простіше, ніж дебажити сервіс із десятком залежностей.
4.  **Читабельність:** Дивлячись на дерево файлів, розробник відразу бачить список доступних дій: `CreateSession`, `CancelTicket`, `FindAvailableSeats`. Щоб знайти необхідний Use Case, йому не потрібно ритися в купі незмістованого коду в одному класі, шукаючи необхідні методи. У великих проектах це дійно стає проблемою. 

---

### 3. Недоліки CQS: що саме ускладнилося?

* **Вибух кількості файлів:** Замість одного файлу сервісу з'являється 10+ файлів для команд, запитів та їхніх хендлерів. Це може бути доволі моторно декомпозувати.
* **Складність проходження даних:** Тепер шлях запиту виглядає так: `Controller -> Command -> Bus -> Handler -> Domain`. Це збільшує кількість абстракцій.
* **Навігація:** Пошук логіки вимагає більше часу, оскільки код "розмазаний" по багатьох дрібних класах. Якщо це звичайні CRUD-операції - то це дійсно може здаватися надмірним ускладненням

---

### 4. Різниця між Command/Query Handler та Service

| Критерій | Класичний Service | Command / Query Handler |
| :--- | :--- | :--- |
| **Відповідальність** | Багато методів для однієї сутності | Один метод (Use Case) на один клас |
| **Залежності** | Тягне всі репозиторії, потрібні для сутності | Тільки ті залежності, що потрібні для конкретної дії |
| **Взаємодія** | Прямий виклик методу класу | Відправка об'єкта через шину (Bus) |

Сервіс "знає забагато", тоді як хендлер фокусується лише на успішному виконанні однієї бізнес-операції.

---

### 5. Вплив CQS на розширюваність

CQS робить систему **адитивною**, а не **модифікативною**:

* **Додавання нової команди:** Щоб додати функцію "Архівувати сеанс", ви просто створюєте новий `ArchiveSessionCommand` та його хендлер. Ви **не чіпаєте** існуючий код. Це мінімізує ризик зламати старі функції (принцип Open/Closed з SOLID).
* **Зміна сервісу:** У класичному підході вам довелося б лізти в `SessionService`, додавати нові залежності в конструктор та змінювати існуючий клас, що часто веде до помилок та незручностей, особливо якщо над одним модулем працює декілька людей.

### 6. Структура даних: Query та Domain Model

У CQS підході Query зазвичай ігнорує та обходить доменний рівень, викликаючи репозиторій напряму та маппячи його в DTO. Це зроблено в першу чергу для оптимізації. Доменний рівень існує для зберігання бізнес-правил та захисту даних, особливо коли ми змінюємо їх (наші commands). У той же час єдина задача Query - це візуалізація даних (UI), він просто **зчитує** дані та відправляє їх клієнту. Нерідко клієнту необхідно надати інформацію, яка потребує доступ до сутностей з інших агрегатів. Тоді для кожного такого джойну нам необхідно буде лізти в інший репозиторій, щоб отримати певні дані. А якщо проект - це модульний моноліт, в якому контексти ізольовані, а комунікація відбувається тільки по певним контрактам? Тоді звичайна Query-операція перетворюється в дуже складний та нечитабельний код, який до того ж тормозить систему. Саме тому, з метою уникнути такої ситуації, ми свідомо порушуємо принципи DDD та Low Coupling, обходячи доменні сутності та внутрішню ізоляцію, для більш простого та оптимального варіанту. Так, архітектура стає більш св'язаною, однак це стосується лише операцій читання, й навіть якщо з'явиться помилка - нічого критичного для системи не буде(хоча на цьому етапі вже буде більш доречно перейти на окрему view-таблицю або взагалі БД, тобто перейти на повноцінний CQRS). В цьому й полягає ключовий сенс використання розділення Command/Query в подібних архітектурах.

# Lab2: Analysis and Comparation

### Що змінилося в структурі проєкту порівняно з лабораторною 1?
Раніше проєкт був розбитий на модулі сутностей (наприклад, `users/`, `sessions/`), де контролер, сервіс та сутність TypeORM лежали поруч. Тепер структура стала **горизонтальною** та чітко розмежованою:

* **Presentation Layer:** Контролери, DTO, Exception Filters. Тільки вхідні точки та валідація запитів.
* **Application Layer:** Сервіси, сценарії використання (Use Cases), порти зовнішніх сервісів. Тут живе **оркестрація** доменних сутностей та функцій, але не сама бізнес-логіка.
* **Domain Layer:** Серце системи. Тільки бізнес-правила, інтерфейси(порти) репозиторіїв та доменні сутності (Entity/Value Objects), які очищені від знань про конкретну базу даних.
* **Infrastructure Layer:** Реалізація репозиторіїв, робота з TypeORM, маппери та функціонал зовнішніх сервісів
* **Core Module:** Модулі, конфіги, підключення інфраструктури та конфігурація проекту відповідно до фраймворку NestJs. Тепер інтерфейс та синтаксис NestJS не змішується з іншою частиною проекту та лежить в окремому модулі

### 2. Які переваги дало розділення на шари?

1.  **Ізоляція бізнес-логіки:** Домен більше не залежить від того, як працює TypeORM, RESTAPI чи NestJs. Тепер доменний шар - це стабільна та ізольована частина коду, яка протягом усієї розробки може не змінюватися, слугуючи "мозком" для всієї системи.
2.  **Тестованість:** Завдяки DIP ми легко підміняємо реальні репозиторії на "моки" (mocks) в Unit-тестах, а щоб перевірити бізнес-правил - ми просто створюємо новий екземпляр доменної сутності та тестуємо її.
3.  **Зменшення зв'язності (Coupling):** Зміни в схемі таблиці БД тепер впливають тільки на інфраструктурний шар, не зачіпаючи всю іншу логіку системи.
4.  **Чіткість відповідальності:** Кожен клас тепер виконує одну роль (Single Responsibility), що полегшує дебаг.

### 3. Які недоліки та ускладнення з'явилися?

* **Велика кількість коду:** Тепер для простого CRUD потрібно розбивати весь код на декілька шарів, для кожного агрегата створювати доменну сутність зі своїми Local Entities та ValueObjects. Також до нього необхідно окремо прописувати інтерфейси до репозиторію, його реалізацію та мапер.
* **Складність мапінгу:** Необхідність постійно переганяти дані з `TypeORM Entity` в `Domain Entity` і назад. Це створює постійне навантаження та ускладнює написання коду.
* **Складна навігація:** Щоб зрозуміти, як працює один метод, доводиться стрибати між 3-4 папками (Presentation -> Application -> Domain -> Infrastructure).
* **Поріг входження:** Умовному новачку буде набагато важче розібратися в структурі такого проекті, ніж у простій примитивній структурі з першої лабораторної.

### 4. Наскільки простіше тепер змінити БД або фреймворк?

* **Зміна БД (наприклад, Postgres на MongoDB):** Значно простіше. Заміть того щоб переписувати увесь код повністю, достатньо просто реалізувати новий репозиторій в шарі `Infrastructure`, який слідує інтерфейсу з шару `Domain`. При цьому `Application` та `Domain` шари взагалі не чіпаються.
* **Зміна фреймворку (наприклад, NestJS на Fastify або Express):** Простіше, але все одно потребує зусиль. Зміни значно торкнуться шару `Presentation` (контролери, декоратори, гуарды). Також через особливість NestJS прийдеться частково модифікувати слої `Application` та `Domain`, але основна логіка використання залишаться недоторканою.

### 5. Чому було обрано саме таку Domain Model?

При використанні Anemic Model ми фактично переносимо перевірку бізнес-правил на шар `Application`, що, як мені здається, є антипатерном. Головна задача `Application` - це оркестрація процесу, тим часом як об'єкти з `Domain` повинні містити усю логіку та правила системи. В цьому головна суть усіх доменно-центричних архітектур. Тому вирішено було чітко слідувати принципам DDD та реалізувати саме Rich Model для доменних сутностей

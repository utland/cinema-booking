# Lab3: Synchronous and asynchronous communication between components. Analysis and Comparation

---

### 1. Що змінилося порівняно з Лабораторною №3?

У попередній лабораторній ми розділили команди та запити (CQS), але виконання все одно залишалося послідовним у межах одного потоку. Тепер:
* **Події (Events):** Замість того, щоб Command Handler викликав інші сервіси напряму, він публікує подію (наприклад, `TicketPaidEvent`).
* **Підписники (Listeners/Subscribers):** З'явилися окремі класи, які "слухають" ці події та виконують додаткові дії.
* **Event Bus:** Впроваджено механізм розсилки подій через вбудований у NestJS механізм бібліотеки `@nestjs/cqrs`.

---

### 2. Виділені побічні операції та обґрунтування

* **Надсилання Email/Push-сповіщень (NotificationService):** Користувач не повинен чекати декілька секунди, поки поштовий сервер відправить квиток, щоб побачити статус "Успішно", адже механізм комунікації реалізований через події.
* **Оплата транзакції:** використовуємо побічний сервіс `PaymentService` з визначеним контрактом. Комунікація обрана синхронна через прямий виклик порта сервісу, адже операція є критичною та вимагає обробки помилок.

---

### 3. Порівняння синхронного та асинхронного підходів

| Критерій | Синхронний підхід | Асинхронний підхід |
| :--- | :--- | :--- |
| **Час відповіді API** | Довший (включає час виконання всіх підзадач). | Мінімальний (відповідь одразу після основної дії). |
| **Поведінка при збоях** | Помилка в payment-сервісі може скасовує всю операцію оплати квитка. | Помилка в email не впливає на успішність бронювання. |
| **Зв'язаність (Coupling)** | Висока: Handler знає про всі побічні сервіси. | Низька: Handler знає тільки про факт події. |
| **Складність** | Просто реалізувати та дебажити. | Важче відстежити ланцюжок подій та стан системи. |

---

### 4. Вибір для Production-системи

В реальній системи зазвичай використовується гібридний режим, в залженості від вимог:
* **Синхронно:** Операції, що потребують негайної перевірки (списання грошей, перевірка вільного місця в залі).
* **Асинхронно:** Все, що стосується сповіщень, аналітики та логування.
* **Обґрунтування:** Це забезпечує найкращий UX (швидка відповідь) та високу відмовостійкість (система працює, навіть якщо впав сервіс статистики).

### 5. Проблема дублювання подій та Ідемпотентність

Якщо подія (наприклад, `PaymentConfirmed`) буде доставлена двічі (через збій мережі або ретраї брокера):

* **Ризик:** Повторне надсилання листа про успішність транзакції.
* **Ідемпотентність обробників:** В ідеалі усі події в системі повинні бути ідемпотентними, особливо якщо стосуються роботи з базою даних. В таких випадкам обробка ідемпотентості критична, оскільки повторное виконання операціїї в бд може обернутися катастрофою для всіє системи. Однак в даному випадку єдиний підписник в події - це NotificationService, який просто надсилає лист про успішну оплату. Ліпити окрему таблицю для подій заради цього - мені здається перебільшенням. Тому обробки ідемпотентності для цього конкретного випадку не реалізовано.
